## 1. 데이터 모델링 및 ERD

### 1.1. 최종 ERD 구조

- 외부 ID와 내부 대리키를 분리
  - source 가 다른 데이터셋에서 id 중복을 방지하기 위함

```mermaid
 erDiagram
        %% Core Domain
        PLACE ||--|| PLACE_BF_INFO : "1:1 (무장애 메타데이터)"

        %% Indoor Map Domain
        PLACE ||--o{ FLOOR_MAP : "1:N (층별 벡터 도면)"
        FLOOR_MAP ||--o{ FACILITY_NODE : "1:N (실내 편의시설 노드)"

        %% Interaction Domain
        FACILITY_NODE ||--o{ REPORT : "1:N (상태 제보 및 PDR 보정점)"
        MEMBER ||--o{ REPORT : "1:N (제보 작성자)"

        %% Help Matching (Archive Log)
        PLACE ||--o{ HELP_MATCHING_LOG : "1:N (발생 장소)"
        MEMBER ||--o{ HELP_MATCHING_LOG : "1:N (요청자 및 도우미)"

        MEMBER {
            bigserial id PK
            varchar role "USER, ADMIN 등"
            varchar nickname
            timestamp created_at
        }

        PLACE {
            bigserial id PK
            varchar external_id "contentid 또는 wfcltId"
            varchar source "KNTO, SSIS, USER"
            varchar category "관광지, 숙박, 공공기관 등"
            varchar name
            varchar sanitized_address "교차 검증용 정규화 주소"
            geometry location_point "Point (공간 쿼리용 야외 좌표)"
            varchar thumbnail_url
        }

        PLACE_BF_INFO {
            bigint place_id PK,FK
            jsonb bf_details "파싱된 JSON 메타데이터"
            timestamp last_synced_at
        }

        FLOOR_MAP {
            bigserial id PK
            bigint place_id FK
            int floor_level "층수 (기압계 데이터와 매칭)"
            jsonb geojson_data "Mapbox용 렌더링 데이터 (feature_id 포함)"
            bigint created_by FK
            timestamptz created_at "BaseAuditEntity"
            timestamptz updated_at "BaseAuditEntity"
        }

        FACILITY_NODE {
            bigserial id PK
            bigint floor_map_id FK
            varchar target_feature_id "★ GeoJSON 폴리곤 고유 ID 매핑"
            varchar node_type "ELEVATOR, TOILET 등"
            varchar name "예: 1층 장애인 화장실"
            geometry geojson_point "실내 절대 Point 좌표 (Snap 기준점)"
            boolean is_checkpoint "PDR 영점 조절 가능 여부"
            int snap_radius "보정 허용 반경(m)"
            bigint created_by FK
            timestamptz created_at "BaseAuditEntity"
            timestamptz updated_at "BaseAuditEntity"
        }

        REPORT {
            bigserial id PK
            bigint node_id FK
            bigint reporter_id FK
            varchar issue_type "BROKEN, REPAIRED 등"
            varchar description "상세 설명"
            timestamp created_at "★ PDR 영점 조절(Snap) 기준 시각"
        }

        HELP_MATCHING_LOG {
            bigserial id PK
            bigint place_id FK
            bigint requester_id FK "도움 요청자"
            bigint helper_id FK "도움 제공자"
            jsonb last_known_location "요청 발생 위치 스냅샷 (floor, lat, lng)"
            timestamp requested_at "최초 요청 시각"
            timestamp completed_at "도움 완료 시각"
        }
```

### 1.2. 무장애 정보 JSONB 스키마 구조

PostgreSQL의 JSONB 타입의 장점을 극대화하기 위해, 장애 유형 및 사용자 목적에 따라 카테고리화된 구조를 채택합니다. 검색의 핵심인 `is_available`을 최우선 필드로 둡니다.

## 2. 인덱스 및 성능 최적화 포인트

Upsert 트랜잭션 자체의 부하는 미미하나, Transform 단계의 **공간 쿼리 및 문자열 유사도 비교**에서 발생할 수 있는 병목을 사전에 차단합니다.

- **공간 인덱스**: `PLACE` 테이블의 `location_point` 컬럼에 `GIST` 인덱스 생성.
- **문자열 인덱스**: `PLACE` 테이블의 `name` 컬럼에 `pg_trgm` 확장을 이용한 `GIN` 인덱스 생성.
- **JSON 인덱스**: `PLACE_BF_INFO` 테이블의 `bf_details` 컬럼 내 주요 필드(`is_available`)에 대해 GIN 인덱스 생성 고려 (빠른 조건 검색용).

  ```java
      {
        "type": "FeatureCollection",
        "features": [
          {
            "type": "Feature",
            "properties": { "name": "복도", "color": "회색", "node_id": id1 },

          },
          {
            "type": "Feature",
            "properties": { "name": "화장실 공간", "color": "흰색", "node_id": id3 },
            "geometry": { "type": "Polygon", "coordinates": [...] }
          }
        ]
      }
  ```

  - 특정 

    ```
    feature
    ```

     의 

    ```
    ${properties.node_id}
    ```

     와 

    ```
    FACILITY_NODE.target_feature_id
    ```

     는 FK로 연결됨

    - 렌더링되는 뷰는 `FLOOR_MAP.geojson_data`의 feature로 표현하고

    - 실제 데이터소스는 

      ```
      FACILITY_NODE
      ```

       로 관리됨

      - `FACILITY_NODE.node_type` 에 따라서 보여지는 데이터의 유형을 변동시킬 수 있음 (확장성 포인트)
      - 또한 추후에 `PDR 초점계산`에서 ‘면’으로 표현된 시설을 대표 위치인 `point` 로 매핑시키기 위한 `geojson_point` 로 매핑 (⇒ 백그라운드 계산을 위한 키)
